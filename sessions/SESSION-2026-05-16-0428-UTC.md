OPEN THREADS -- TIMESTAMP: 2026-05-16 04:28 UTC

1. PERSONS DEDUP BUG (CRITICAL):
   scripts/import-ftm.mjs:405-407 fetches existing FTM persons with
   `.select('id, ancestry_id').like('ancestry_id', 'ftm:%')`. PostgREST caps
   the response at 1,000 rows. With 1,576 FTM persons in the synced tree,
   the other 576 are seen as "new" on every run and inserted with fresh UUIDs
   (persons.ancestry_id has no unique constraint, so duplicate values are
   accepted). This bug is invisible on the test file (144 persons) and on the
   first run of a large tree (existing fetch returns 0). It manifests on the
   2nd+ run of any tree larger than 1,000 persons.

   Fix shape (2-line change): paginate the existing fetch -- repeat with
   `.range(offset, offset+999)` until response is empty, accumulating into
   existingIds. Or: switch to a server-side RPC that returns all rows.

   Alternative fix: add a unique constraint on persons.ancestry_id and use a
   real upsert. Slightly bigger schema change, but eliminates the manual
   select+update/insert dance entirely.

2. CLEANUP STRATEGY for the persons table:
