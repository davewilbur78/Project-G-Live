#### Q9 — Supabase .in() limit risk
**Confirmed risk, code already anticipates it.** `src/app/api/ftm-import/route.ts:34` has a NOTE comment: *".in() with >500 UUIDs may need chunking for the full ~1500-person tree."* Three .in() calls follow at lines 67, 73, 80 against a list of 1,576 person UUIDs.

Supabase PostgREST passes .in() values via URL query string. 1,576 UUIDs × 37 chars + commas ≈ 58 KB URL — well past typical 8-16 KB URL caps. Will either truncate, error out, or silently miss rows.

**The import script itself (scripts/import-ftm.mjs:531) DOES chunk to 200**, so the import is safe. The leak is on the **stats / read endpoint** (route.ts), which the brief's Q9 specifically asked about.

**Recommended fix (out of scope this session):** chunk those 3 .in() calls to ~200 IDs per call, or use a server-side RPC that takes the list as JSON and filters in SQL.

The dry-run intentionally skips `sourceIdMap` population (line 463 wraps in `!DRY_RUN`), so the wiring count in dry-run will *always* report 0. Real-run wiring works perfectly. Lesson: read the code path before alarming.

2. **PersonExternal and MediaFile aren't extracted** — Module 17 Phase 2 marked complete in AGENT.md, but those two tables were never wired into the extractor. Not a bug per se (Phase 2 didn't promise them), but worth flagging that "we have the synced tree" doesn't yet equal "we have the synced data" for those two domains.
