THE DECISION MADE THIS SESSION
The genealogical data layer of Project-G-Live -- persons, families, events, sources,
citations, places, repositories, associations -- was built migration by migration without
a complete picture of where it needed to end up. We now have that picture (TNG's
battle-tested schema + our GPS additions). The foundation should be rebuilt properly before
any more modules are built on top of it. This is worth doing now while the database is
still young.

WHAT THE NEXT SESSION MUST DO

This is a FIX session. Possibly two sessions. No new modules. No new UI. Foundation only.

The work, in priority order:

1. PERSONS TABLE -- bring it up to what it should have been from day one:
   - Name components: lnprefix, suffix, title, prefix (name prefix), nickname
   - Sex field
   - Living flag (boolean)
   - Private flag (boolean)
   - Dual-date pattern on birth and death: birth_date_display (varchar) + birth_date_sort (date),
     death_date_display (varchar) + death_date_sort (date)
   - Rename birth_year/death_year to something that accommodates the dual-date pattern
   - changedate as proper timestamp (it probably already is -- verify)
   - changedby field

2. FAMILIES TABLE -- does not exist at all. Build it:
   - familyID, partner1_id (FK persons), partner2_id (FK persons)
   - marriage_date_display, marriage_date_sort, marriage_place
   - marriage_type (MARR/MARB/MARL/etc.)
   - div_date_display, div_date_sort, div_place
   - living, private flags
   - Note: use partner1/partner2 not husband/wife -- we are not replicating GEDCOM's
     binary sex assumption. This is one place where we go further than Darrin.

3. FAMILY_MEMBERS TABLE (TNG's tng_children) -- bridge table:
   - person_id (FK), family_id (FK)
   - role (child / partner -- separates children from spouses in the family record)
   - relationship_to_partner1 (natural/adopted/step/foster/unknown)
   - relationship_to_partner2 (same)
   - birth_order
   - has_descendants (boolean, denormalized for display)

4. REPOSITORIES TABLE -- does not exist at all. Build it:
   - repository_id, name, type (archive/library/online/courthouse/church/other)
   - url, address_line, city, state, country
   - notes
   - Simple. Clean. TNG's pattern: delegate address detail here, not on the source.
   - After building: add repository_id FK to sources table (sources already exist)

5. ASSOCIATIONS TABLE (TNG's tng_associations) -- does not exist at all:
   - association_id, person_id (FK), associated_person_id (FK)
   - association_type (witness/godparent/employer/neighbor/colleague/other)
   - description (free text)
   - source_id (FK -- what record established this association)
   - date_display, date_sort (when was the association documented)
   - This IS the FAN Club data model. Module 8 builds on this.

6. EVENT TYPES TABLE (formalizing what we have):
   - event_type_id, tag (RESI/OCCU/IMMI/BIRT/etc.), display_name
   - scope (individual/family -- our version of TNG's I/F type)
   - is_built_in (boolean -- marks the standard GEDCOM types)
   - sort_order
   - Populate with standard types at migration time.
   - Add event_type_id FK to timeline_events table.

7. DUAL-DATE RETROFIT on existing tables:
   - timeline_events: event_date is currently text. Add event_date_sort (date).
   - Any other date fields across the schema that are not already dual-dated.

8. VERIFY AND DOCUMENT the current schema state before writing any migrations:
   - Pull docs/architecture.md from the repo and confirm what currently exists
   - List every table and every field we already have
   - Then write migrations only for what is missing or needs changing
   - DO NOT assume the current state from memory -- read it from the repo
